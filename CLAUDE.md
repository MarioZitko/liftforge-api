# LiftForge API — CLAUDE.md

## Project overview

NestJS 11 REST API for LiftForge, a coaching platform that lets coaches create training programs for clients. Uses PostgreSQL via Prisma as the data layer. Deployed to a Hetzner VPS via Docker.

## Tech stack

| Layer | Technology |
|---|---|
| Framework | NestJS 11 (Express platform) |
| Language | TypeScript 5 |
| ORM | Prisma 6 with PostgreSQL |
| Auth | JWT (access + refresh tokens via HTTP-only cookies), Passport.js |
| OAuth | Google and Facebook via passport-google-oauth20 / passport-facebook |
| Email | Resend |
| Rate limiting | @nestjs/throttler (10 req / 60s per IP) |
| Request-scoped storage | nestjs-cls |
| Validation | class-validator + class-transformer |
| API docs | Swagger at `/api` |
| Testing | Jest + Supertest |

## Domain model summary

- **User** — base identity with `Role` enum: `CLIENT | COACH | ADMIN`
- **Client** / **Coach** — role-specific profiles extending User (one-to-one)
- **Program** → **TrainingBlock** → **TrainingWeek** → **Training** → **TrainingExercise** — the hierarchical training structure
- **ClientProgram** — assigns a Program to a Client (with coach, start/end date, status)
- **Exercise** — global exercise library with primary/secondary muscles
- **Volume** — computed total volume per TrainingExercise
- **Invitation** — coach sends invite tokens to onboard clients
- All models carry `createdBy` / `updatedBy` audit fields (soft deletes on User via `deletedAt`)

## Authentication flow

1. **Login** → issues JWT access token (1h, Bearer) + HTTP-only refresh token cookie
2. **Refresh** → `POST /auth/refresh` exchanges cookie for new access token
3. **Guard** → `JwtAuthGuard` validates Bearer token on protected routes
4. **Roles** → `RolesGuard` + `@Roles()` decorator enforces `CLIENT | COACH | ADMIN`
5. **CLS** → `ClsUserMiddleware` stores `userId` from the JWT payload in CLS after Passport resolves it; services read `cls.get('userId')` for audit fields
6. **OAuth** → Google and Facebook strategies redirect to `/auth/google` and `/auth/facebook`; callback at `/auth/google/callback` etc.
7. **Email verification** — required before login is permitted; token stored in `VerificationToken`
8. **Password reset** — token stored in `PasswordResetToken`, emailed via Resend

## Response envelope

Every successful response is wrapped by `ResponseInterceptor`:
```json
{ "statusCode": 200, "timestamp": "...", "data": { ... } }
```

Errors are caught by `AllExceptionsFilter`:
```json
{ "statusCode": 400, "timestamp": "...", "path": "/auth/login", "message": "..." }
```

## Prisma workflow

The schema is split into per-domain files under `prisma/models/` and merged into `prisma/schema.prisma` before generation.

```bash
npm run prisma:merge      # merge models/ → schema.prisma
npm run prisma:generate   # merge + generate Prisma client
npm run migrate           # prisma migrate dev (development)
npm run seed              # seed development DB
npm run seed:staging      # seed staging DB
```

Always run `prisma:generate` after pulling schema changes. The generated client lives in `generated/prisma/`.

## Adding a new feature module

1. Create `src/modules/<name>/` with `<name>.module.ts`, `<name>.controller.ts`, `<name>.service.ts`, DTOs in `dto/`
2. Import `PrismaModule` if needed (it's global, but declare the import for clarity)
3. Register the controller + service in `app.module.ts` (or create a proper `@Module` and add it to `imports`)
4. Add Prisma models to `prisma/models/<name>.prisma`, then run `npm run prisma:generate`
5. Protect routes with `@UseGuards(JwtAuthGuard, RolesGuard)` + `@Roles(Role.COACH)` as appropriate

## Common patterns

**Protecting a route:**
```typescript
@UseGuards(JwtAuthGuard, RolesGuard)
@Roles(Role.COACH)
@Get()
findAll() { ... }
```

**Getting the current user in a controller:**
```typescript
@CurrentUser() user: JwtPayload
```

**Getting userId in a service (via CLS):**
```typescript
constructor(private cls: ClsService) {}
const userId = this.cls.get('userId');
```

**Standard DTO pattern:**
```typescript
export class CreateExerciseDto {
  @IsString()
  @IsNotEmpty()
  name: string;
}
```

## Module registration note

Some modules (program, client-program, training-*, training-exercise) are registered directly in `AppModule` as loose controllers/providers rather than as self-contained `@Module` classes. This is intentional for now; when extracting to proper modules, move them out of `app.module.ts`.

## Testing

- Unit tests live next to the files they test (`*.spec.ts`)
- E2E tests in `test/` with separate `jest-e2e.json` config
- Use `@nestjs/testing` `Test.createTestingModule()` for unit tests
- Use real DB for integration tests — do not mock Prisma in tests that exercise DB logic

## Docker deployment (Hetzner)

The API runs as a Docker container alongside a PostgreSQL container and an nginx container (serving the frontend + proxying `/api`). Key considerations:
- Run `prisma migrate deploy` (not `migrate dev`) on container startup in production
- Read env vars from Docker secrets or an env file mounted at runtime
- The built output is `dist/main.js`; production start command is `node dist/main`
